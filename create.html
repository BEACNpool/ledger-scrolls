Put a file on Cardano forever.

A scroll is any file written straight onto the blockchain — no server, no IPFS, no link that can rot. Your reader rebuilds the exact bytes from chain data and checks the hash. This page covers what it costs, how the bytes are packed so a reader can recover the maximum data per transaction, and the steps to mint one.

What is a scroll?

An immutable file stored in the Cardano ledger itself. There are two ways the bytes live on-chain:

  • Standard Scroll (LS-LOCK): the whole file sits in one transaction's inline datum, locked at an "always-fail" script address that can never be spent. One query to read. Best for tiny files (≤~14 KB).
  • LS-CHAIN v2: the file is split across many plain "page" transactions (no NFTs, nothing locked per page), anchored by a single manifest datum that lists them in order and carries the hashes. This is the format for anything larger — documents, audio, video, datasets.

A reader fetches the manifest, pulls the listed pages, concatenates them, verifies every hash, and — only if it all matches — renders the file. Nothing is trusted that isn't proven against a hash.

An older format, CIP-25 "pages" (LS-PAGES), mints one NFT per page. Every legacy scroll is still read that way, but it costs ~6× more and locks ADA in an NFT per page — don't use it for new scrolls.

What it costs — ADA per KB

On-chain storage is priced by the byte. Two components:

  • A one-time floor of ~1.5–2 ADA per scroll — the locked manifest (LS-CHAIN) or the locked datum UTxO (Standard). Doesn't grow with file size.
  • ~0.06 ADA per KB of content on top (transaction fees for the page txs).

Rule of thumb: ADA ≈ 1.5 + (fileKB × 0.06). The three formats compared (mainnet params, 2026):

Pick a format

  ≤ ~14 KB after gzip   →  STANDARD SCROLL   (one locked UTxO; simplest, strongest)
  anything larger       →  LS-CHAIN v2       (bare metadata pages + manifest)

That's the whole decision. If gzip gets your file under ~14 KB, a Standard Scroll is the strongest guarantee the protocol offers. Otherwise reach for LS-CHAIN v2.

How the data is packed (so a reader recovers the maximum)

This is the part that decides how much file you fit per transaction. The Cardano ledger has two hard limits you build around:

  • Each metadata byte string is capped at 64 bytes. So your file's bytes are chopped into 64-byte segments (128 hex characters each).
  • Each transaction is capped at 16,384 bytes. So segments are grouped into pages sized to sit safely under that limit.

The recommended packing — 190 segments × 64 bytes = 12,160 bytes of file per page transaction — keeps each tx comfortably under 16 KB with room for the rest of the transaction. That is the "maximum data per NFT/tx" number: pack to 190×64 and you minimize the page count (and therefore the fees) for any file.

What one page transaction carries

Each page is a plain self-send transaction with metadata under label 22025:

{ "22025": {
    "v": 2,                 // format version
    "i": 1,                 // 1-based page index
    "n": 47,                // total pages
    "sha": "0x<32 bytes>",  // SHA-256 of THIS page's payload
    "p": [ "0x<64B>", "0x<64B>", … ]  // up to 190 segments
} }
your file → gzip (if it helps) → hex
  ↓ split into 64-byte segments
page 1  [ seg 1seg 2…seg 190 ]  = 12,160 B
page 2  [ seg 191…seg 380 ]  = 12,160 B
  …
manifest [ page 1 txhashpage 2 txhash… ] + hashes + content-type

A page's payload is just its segments concatenated in order. The whole file's encoded stream is the page payloads concatenated in manifest order — so page boundaries are arbitrary; i and n are integrity aids.

The manifest — the index that makes it parseable

The manifest is one locked inline datum (valid Plutus Data, Constr 0) that lets a reader rebuild the file deterministically in one query, with no guessing:

version        // 2
contentType    // MIME, e.g. "video/mp4"
codec          // "gzip" | "none"
sizeDecoded    // byte length of the original file
sha256Decoded  // hash of the original file
sha256Encoded  // hash of the encoded (gzipped) stream
pageTxHashes   // ordered list of page transaction hashes
next           // link to a continuation manifest, or none

Because pages are listed by explicit transaction hash, a reader never has to scan a policy or guess which tx is which — it fetches exactly the listed pages, batched, and verifies each against the manifest's hashes. (One manifest holds ~400 page hashes ≈ ~4.8 MB; bigger files chain manifests via next.)

How a reader rebuilds it

  1. Read the manifest datum (one UTxO query).
  2. Collect pageTxHashes (following next links).
  3. Batch-fetch the page metadata (label 22025) for all those tx hashes.
  4. Concatenate each page's segments, in manifest order → the encoded stream. Verify each page's sha and the stream's sha256Encoded.
  5. If codec = gzip, gunzip. Verify sha256Decoded.
  6. Only then render, by contentType (HTML is always fully sandboxed).

Optimize your file (this is where the ADA is)

Cost is bytes, so shrink the file before it touches the chain, then declare the encoding honestly. codec: gzip helps only uncompressed data (text, HTML, SVG, JSON); media is already compressed — declare none.

Mediacontent_typecodecTips
Text / Markdowntext/plain; charset=utf-8gzip if >~8 KBUTF-8, \n endings
Imageimage/png, image/jpegnoneoptimize, strip EXIF
SVGimage/svg+xmlgzipminify; no scripts/external refs
HTML doctext/htmlgzipself-contained, no JavaScript
Audioaudio/ogg; codecs=opusnoneOpus 24–32 kbps speech, 64–96 music
Video (+ audio)video/mp4 / video/webmnoneone file, both tracks — see below

Audio and video together is just one file — a normal MP4 or WebM with both a video and an audio track and a single content_type. The viewer plays picture and sound from one element; there is no separate "audio scroll."

Video: pick your codec by priority

Maximum future-proof (plays anywhere) — H.264 + AAC:

ffmpeg -i in.mov -c:v libx264 -crf 30 -preset veryslow \
  -vf scale=-2:480 -c:a aac -b:a 64k -movflags +faststart out.mp4

Smallest bytes (~30–50% less, plays in modern browsers) — AV1/VP9 + Opus:

ffmpeg -i in.mov -c:v libsvtav1 -crf 38 -preset 5 \
  -vf scale=-2:480 -c:a libopus -b:a 64k out.webm

The real levers are resolution, duration and bitrate — not the container. Avoid HEVC/H.265 (patent + spotty playback). Codec choice alone roughly doubles your seconds-per-ADA.

Mint it — the workflow

1 · Hash your original

sha256sum myfile        # record this — your permanent commitment

2 · Rehearse on testnet (free)

The bytes are identical on Preview testnet; the mistakes are free. Run the full pipeline there first.

3 · Mint

Standard Scroll (tiny files):

cd scripts
./mint-standard-scroll.sh myfile payment.skey payment.addr

LS-CHAIN v2 (everything larger):

# prepare: gzip-if-it-helps, hash, split into 190×64-byte pages
python3 tools/lschain/prepare.py myfile --out build/
# mint the page txs, then build + lock the manifest datum
tools/lschain/mint.sh build/ payment.skey payment.addr

4 · Verify from chain (do not skip)

cd koios-viewer && python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python3 -m lsview reconstruct-chain --txin <MANIFEST_TX>#0 --out check.bin
sha256sum check.bin     # must equal the hash from step 1

5 · Register & share

Add a registry entry with your pointer and the required sha256, open a PR to the repo, and post it with #LedgerScrolls:

{ "name": "my-scroll",
  "pointer": { "kind": "manifest-chain-v2", "txHash": "<manifest tx>", "txIx": 0 },
  "contentType": "video/mp4",
  "sha256": "…",
  "description": "…" }

The registry is forkable — you can run your own head or get listed in BEACN's public one.

Three rules before you spend a single ADA

1 · It is permanent and public. No edit, no delete, no takedown. Never put in secrets, private keys, personal data, or anything you might regret.

2 · Hash before, verify after. Record the SHA-256 before minting and rebuild it from chain afterward — the hashes must match before you announce anything.

3 · Self-contained only. No external images, fonts, CDNs or IPFS links. A scroll is rendered offline; anything pointing outward is broken forever. Inline everything.
