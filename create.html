⚒ How to make one

Put a file on Cardano forever, in 4 steps.

A "scroll" is just a file written straight onto the blockchain. The whole trick: cut the file into equal pieces, add an index, and write every piece on-chain. Here's the idea in five frames:

🎬Your file
(e.g. movie.mp4)→
✄Cut into equal pieces (~12 KB each)→
🧾Add a manifest — the index→
⛓Every piece written on Cardano→
✅Anyone rebuilds it & checks the hash

1

Check the cost, pick a format

2

Choose your file

Anything that's a sequence of bytes works — a photo, a song, a PDF, a short .mp4. Two things matter:

  • Make it small first. Every byte is ADA. Trim it, lower the resolution, pick an efficient codec. (For a short video: 480p, a low bitrate, H.264 or AV1.)
  • It must be self-contained. No links to outside images, fonts or servers — a scroll is rebuilt offline, so anything pointing outward is broken forever.

Whatever its final size, note it — that's the only number the next step needs.

3

The file is cut into equal pieces

A single Cardano transaction can only carry so much data, so the file is sliced into equal chunks. Each chunk is one transaction on the blockchain.

max data per piece = 12,160 bytes (≈ 12 KB)
number of pieces = file size ÷ 12 KB, rounded up

So a ~115 KB clip becomes about 10 pieces; a 1 MB file, about 90. The calculator above is doing exactly this division for you.

12345678910 ← 10 pieces, ~12 KB each ≈ one 115 KB file

Why 12 KB? Cardano caps each stored value at 64 bytes, so the file's bytes are packed 64 at a time — 190 of those (12,160 bytes) is the most you can fit in one transaction. Packing each piece full is what keeps the number of pieces — and the cost — as low as possible.

4

The first piece is the manifest — the index

One special piece is the manifest. It carries no file data; instead it tells a future reader exactly what to expect:

  • what the file is — e.g. video/mp4;
  • how many pieces there are, and their order;
  • a fingerprint (hash) of every piece and of the whole file.
🧾 manifest points to → 12345…

A reader starts at the manifest, follows its list to grab every piece, glues them back together in order, and checks the fingerprints. If they all match, the file is provably the original — and only then is it shown. That's the whole magic: nothing is trusted that isn't verified against the manifest.

Heads-up on the word "NFT." In the older, pricier format each piece was a minted NFT under one policy and the manifest was the first NFT — the exact picture you may have in mind. The modern format (the Chain Scroll) keeps the same idea but the pieces are plain transactions, not NFTs, so it's ~6× cheaper and locks no ADA. Same story, smaller bill.

Ready to make one?

The whole workflow is: hash your file → rehearse on the free testnet → mint → rebuild it from chain to confirm the hash → share it.

# 1. write down your file's fingerprint
sha256sum myfile.mp4

# 2. cut into pieces + build the manifest, then mint
python3 tools/lschain/prepare.py myfile.mp4 --out build/
tools/lschain/mint.sh build/ payment.skey payment.addr

# 3. rebuild from chain — the hash MUST match step 1 before you announce
cd koios-viewer && python3 -m lsview reconstruct-chain --txin <MANIFEST_TX>#0 --out check.mp4
sha256sum check.mp4

Step-by-step with every command: the Your First Scroll quickstart. Per-media tips (codecs, sizes): the Creating Scrolls guide. The exact on-chain format: the chain-format spec.

Three rules before you spend a single ADA

1 · It's permanent and public. No edit, no delete, no takedown. Never include secrets, private keys, or anything you'd regret being public forever.

2 · Hash before, verify after. Record the fingerprint before minting, rebuild from chain after — they must match before you tell anyone.

3 · Self-contained only. Inline everything. No external images, fonts, CDNs or links — they'll be broken forever.
